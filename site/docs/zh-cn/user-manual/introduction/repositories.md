# 媒体与仓库

::: warning 注意
流明集当前处于 Beta 阶段。请先使用测试媒体或已有可靠备份的资料库进行试用，不要将本应用作为重要媒体的唯一存储位置。
:::

在流明集中，用户首先选择一个**存储根目录**（以下简称 `root`）。**仓库**是流明集在该 `root` 下创建并登记管理的独立子目录。媒体原件存放在仓库中；相册、合集和筛选结果则是用于组织这些媒体的应用数据，不是原件的另一份副本。

## 仓库是什么

首次设置时，流明集会引导你选择 `root`，并强制创建一个位于 `<root>/primary/` 的**主仓库**。当你没有为上传指定其他仓库时，主仓库是默认的存储目标。

之后在应用中新建的普通仓库也位于同一个 `root` 下。例如，名为“家庭”的仓库会使用 `<root>/家庭/` 这样的独立子目录。当前的 Desktop 与 Docker 版本都使用这一布局：应用内新建仓库时，不能选择 `root` 以外的独立路径。

```text
<root>/
├─ primary/       # 强制创建的主仓库
├─ 家庭/          # 用户新建的普通仓库
└─ 旅行/          # 用户新建的普通仓库
```

每个由流明集初始化的仓库都有自己的稳定标识和仓库配置。流明集还会在仓库内维护一个内部管理区域，用于暂存、非破坏性编辑记录等应用数据。请不要手动删除或修改这些管理文件。

::: info 仓库不等于数据库
数据库记录媒体的索引、元数据和组织关系；仓库保存媒体文件。两者需要保持对应，但它们承担的职责不同。
:::

## 媒体如何进入仓库

| 方式 | 流明集的处理 | 对来源文件的影响 |
| --- | --- | --- |
| 上传 | 先将内容写入仓库内的暂存区，再提交到正式存储位置 | 在仓库中产生一份由流明集管理的文件；不删除上传端的来源文件 |
| 扫描 | 读取仓库目录中已有的支持文件，计算内容指纹并建立资源记录 | 原位登记，不因扫描而移动已有文件 |
| 云端导入 | 将云端媒体下载到本地暂存区，再提交至仓库 | 在本地仓库中保存下载内容；是否影响云端原件取决于云端服务，流明集的导入流程不会主动删除云端原件 |

上传或导入遇到同名文件时，具体行为由仓库的同名文件策略决定。建议在试用阶段保持默认的**自动重命名**策略，避免覆盖已有文件。

## 移除仓库会发生什么

从流明集中移除仓库，只会解除该仓库在当前实例中的登记；**不会删除仓库目录或其中的媒体**。只要仓库仍位于当前 `root` 下，且目录和配置完整有效，之后可以将它重新登记。当前版本不支持通过新建或重新登记绕过 `root` 边界。

这与删除单个资源是不同的操作。关于资源删除、恢复和原件保护的说明，请阅读[数据完整性与备份](./integrity.md)。

## 使用前注意事项

- 在导入、扫描或后台处理进行期间，不要从文件管理器移动、重命名或删除仓库内的文件。
- 运行流明集的系统账户必须能够读取仓库；上传、导入和内部管理功能还需要写入权限。
- 在 Docker 部署中，必须将仓库路径映射到宿主机上的持久化存储位置。仅存在于容器可写层中的媒体，可能随容器替换而丢失。
- 在移动仓库、更换磁盘或调整挂载路径前，先停止相关处理任务并确认已有可用备份。

::: tip 建议的第一次试用方式
先在向导创建的主仓库中放入少量已有备份的照片，完成[首次使用](./first-use.md)中的上传、删除与恢复验证。需要验证多仓库管理时，再在同一 `root` 下新建专用测试仓库；确认行为符合预期后，才逐步增加正式媒体。
:::
