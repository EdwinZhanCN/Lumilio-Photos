# 数据完整性与备份

::: warning 注意
流明集当前处于 Beta 阶段。请先使用测试媒体或已有可靠备份的资料库进行试用，不要将本应用作为重要媒体的唯一存储位置。
:::

流明集通过内容指纹、暂存提交、软删除和非破坏性编辑降低意外覆盖或丢失的风险。这些是**完整性保护措施**，不是对硬件、文件系统或操作失误的绝对保证，也不能替代独立备份。

## 内容指纹

流明集使用 **BLAKE3** 为媒体计算内容指纹。它可以将文件内容映射为一个稳定的标识，用于上传前预检查、识别已存在的相同内容，以及辅助重复项分析。

内容指纹本身不会产生副本，也无法在原件已损坏或丢失后将其恢复。当前文档也不应被理解为流明集会持续巡检磁盘上的全部原件。

## 写入与扫描

上传和云端导入的文件会先进入仓库内的暂存区，再提交到正式存储位置。这使未完成的写入与正常入库的媒体保持分离。

扫描仓库时，流明集会读取已有文件并建立资源记录，不会因扫描而移动原文件。更多区别请参阅[存储位置与仓库](./repositories.md)。

## 非破坏性操作

| 保护措施 | 能够降低的风险 | 不能替代 |
| --- | --- | --- |
| 内容指纹 | 重复内容被当作不同媒体再次入库 | 原件副本或持续磁盘巡检 |
| 暂存后提交 | 未完成写入被当作正常入库文件 | 文件系统和硬件保护 |
| 软删除 | 普通删除后立即失去原件 | 长期备份或未来的永久删除确认 |
| 非破坏性编辑 | 调整操作直接覆盖原始媒体 | 导出文件和原件的备份 |

当前 Beta 版本中，普通删除会将资源标记为已删除，不会立即移除物理文件；从回收站恢复时也不会移动原件。目前尚未提供面向用户的永久删除或回收站自动清理功能；未来若引入这些功能，将需要明确提示其会销毁原件。

工作室中的调整保存为独立的编辑记录，默认不覆盖原始媒体。导出的成品是新文件，不应被当作原件的唯一备份。具体操作请参阅[工作室](../features/studio.md)。

## 你仍然需要备份

一个可恢复的流明集资料库通常同时需要**媒体仓库**和**数据库备份**。只备份数据库无法恢复原始照片；只备份媒体文件则可能丢失相册、人物、标签和其他应用数据。

Desktop 的私有 PostgreSQL 数据目录位于应用数据目录的 `postgres/18/data` 下：macOS 通常是 `~/Library/Application Support/Lumilio Photos/`，Windows 通常是 `%LocalAppData%\Lumilio Photos\`。Docker 配置器使用名为 `db_data` 的 volume 持久化 `/var/lib/postgresql/data`。这些底层目录应由应用或 PostgreSQL 管理，不要在服务运行时直接复制、改名或编辑其中的文件；日常迁移与恢复优先使用应用生成的逻辑备份。

流明集在“设置 → 服务器”中提供 PostgreSQL 数据库的自动备份、立即备份、下载、恢复和删除。Desktop 的备份保存在本机应用数据目录的 `backups/`；Docker 使用 app-state 中配置的独立备份目录。它不会跟随默认或外部存储位置移动。自动备份默认开启、每 24 小时执行并保留最近 14 份；管理员可调整这些设置。备份只包含数据库数据，**不包含媒体原件**。

因此仍需独立备份所有存储位置中的仓库，并在备份前确认没有正在进行的写入任务。Desktop 与 Docker 的数据库运行位置不同，但应用生成的逻辑备份都不能代替媒体副本。如果条件允许，可采用 **3-2-1** 原则：保留至少 3 份数据、使用 2 种不同存储介质，并将其中 1 份存放在其他地点。这是通用备份建议，不代表流明集已自动完成这些副本。

数据库备份、凭据、云端会话和日志属于 app-private state，不位于 `.lumilioroot` 下。备份方案必须分别覆盖媒体仓库、数据库 dump 和恢复所需的应用私有状态；包含凭据的副本应使用加密存储并限制访问。不要把密钥复制到公开共享目录或不受信任的云盘。

### 正式使用前

- 首次试用只使用少量已有可靠备份的照片；此时先确认这些来源副本可读。
- 确认每个存储位置所在磁盘具有足够空间。
- 确认 Docker volume 或 Desktop 默认及外部存储位置指向预期路径。
- 完成初始化和试用验证后，分别备份所有媒体仓库与数据库并确认备份可读，再扩大资料库。

### 验证备份可以恢复

1. 在“设置 → 服务器”选择“立即备份”，等待新条目出现并下载该文件。
2. 将媒体仓库、数据库备份和所需密钥复制到与生产资料库隔离的测试环境，不要直接拿唯一的正式实例试错。
3. 在测试实例恢复数据库，确认管理员可以登录，并核对资源总数、相册、人物、标签和工作室编辑记录。
4. 抽查多个仓库中的原件、缩略图和数据库记录能否正确对应，再记录本次恢复所需的版本、路径和步骤。

仅能下载或解压备份文件不代表它一定可恢复；只有完成恢复并核对应用数据与原件映射，才算形成闭环。

数据库备份的界面操作见[设置](../features/settings.md#数据库备份-仅管理员)。当前文档仍不能替代完整的媒体备份与灾难恢复演练；在确认所有仓库副本和数据库恢复都真实可用前，请只使用原本已有可靠备份的测试媒体，不要据此迁移正式资料库。回收站的当前行为可参阅[工具](../features/utilities.md)。
